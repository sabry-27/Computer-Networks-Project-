# Virtual Networks Lab using Linux and CORE Network Emulator

## Project Overview

This project investigates TCP performance and dynamic routing behavior in a virtual network environment built using Linux and the CORE Network Emulator. The work focuses on how TCP throughput changes under different window sizes, path lengths, link capacities, packet loss rates, and OSPF routing-cost changes.

The project also uses Wireshark packet captures to inspect TCP/IP headers, ACK packets, data payloads, and OSPF control traffic. The final objective is to connect theoretical networking concepts with practical measurements taken from an emulated network topology.

## Main Objectives

- Build and run a virtual network topology using Linux-based nodes.
- Measure TCP throughput and retransmissions using `iperf3`.
- Study the effect of TCP window size on performance.
- Inspect TCP, IPv4, and Data Link layer headers using Wireshark.
- Compare TCP performance over short and long network paths.
- Analyze the impact of capacity and packet loss on TCP throughput.
- Observe OSPF path recalculation after changing interface costs.
- Capture and analyze OSPF Link State Updates after topology changes.

## Tools and Technologies

- **CORE Network Emulator**: Virtual network topology creation and testing.
- **Linux Nodes**: Used as routers, clients, and servers inside the emulated topology.
- **iperf3**: TCP throughput and retransmission measurements.
- **Wireshark**: Packet capture and protocol inspection.
- **traceroute**: Path and RTT observation.
- **OSPF / Quagga CLI**: Dynamic routing configuration and route-table inspection.
- **TCP/IP Protocol Suite**: Main protocol stack analyzed in this project.

## Network Scenario

The project mainly analyzes traffic between:

- **n7**: Client node.
- **n11**: Server node used for the main short-path TCP tests.
- **n8**: Server/host used for long-path comparison.
- **n2, n4, n5**: Router nodes used for OSPF database and cost-change experiments.

The tests were performed using TCP traffic generated by `iperf3`, while routing behavior was controlled and observed through OSPF.

# Project Workflow

## 1. TCP Window Size Experiment

The first experiment studied the effect of TCP window size on throughput and retransmissions between **n7** and **n11**.

`iperf3` was executed for 40 seconds while changing the TCP window size from small values to larger values.

### Main Results

| TCP Window Size | Average Throughput | Retransmissions |
|---|---:|---:|
| 1 KB | 139 Kbits/sec | 0 |
| 2 KB | 362 Kbits/sec | 0 |
| 3 KB | 718 Kbits/sec | 0 |
| 4 KB | 722 Kbits/sec | 0 |
| 5 KB | 720 Kbits/sec | 0 |
| 6 KB | 96.4 Kbits/sec | 278 |
| 12 KB | 449 Kbits/sec | 350 |
| 16 KB | 432 Kbits/sec | 310 |
| 24 KB | 173 Kbits/sec | 329 |
| 32 KB | 147 Kbits/sec | 326 |

### Interpretation

Small window sizes produced stable results because the network buffers could handle the amount of in-flight data. The best performance appeared around **3 KB to 5 KB**, especially at **4 KB**, where throughput reached about **722 Kbits/sec** with no retransmissions.

When the window size reached **6 KB**, the throughput dropped sharply and retransmissions increased. This indicates buffer overflow and packet loss. For larger windows, TCP entered a repeated congestion-control cycle caused by **AIMD**: Additive Increase and Multiplicative Decrease.

In other words, TCP slowly increases its sending rate while the network appears stable, but when packet loss occurs, TCP reduces the congestion window aggressively. This creates throughput oscillation and prevents the sender from fully utilizing the link.

## 2. Wireshark Packet Header Analysis

After generating TCP traffic, Wireshark was used to inspect captured frames between **n7** and **n11**.

### Data Packet Analysis

A TCP data frame sent from **n7 to n11** was selected. The captured frame size was **1516 bytes**.

| Layer | Protocol | Size |
|---|---|---:|
| Transport Layer | TCP | 32 bytes |
| Network Layer | IPv4 | 20 bytes |
| Data Link Layer | Linux SLL cooked capture | 16 bytes |
| Data Payload | Application Data | 1448 bytes |
| **Total** |  | **1516 bytes** |

The TCP header was 32 bytes because it included the normal 20-byte TCP header plus 12 bytes of TCP options. The options were:

| TCP Option | Size |
|---|---:|
| NOP x2 | 2 bytes |
| Timestamp | 10 bytes |

### ACK Packet Analysis

A pure ACK packet sent from **n11 to n7** was also inspected. Its total captured frame size was **68 bytes**.

| Layer | Protocol | Size |
|---|---|---:|
| Transport Layer | TCP | 32 bytes |
| Network Layer | IPv4 | 20 bytes |
| Data Link Layer | Linux SLL cooked capture | 16 bytes |
| Data Payload | None / Pure ACK | 0 bytes |
| **Total** |  | **68 bytes** |

This confirms that ACK packets can carry no application payload while still containing TCP, IP, and Data Link layer headers.

## 3. Short Path vs Long Path Throughput

The next experiment compared TCP throughput between two different paths:

- **n7 → n11**: Short path.
- **n7 → n8**: Long path.

Both tests used a fixed TCP window size of **4 KB**.

| Connection | Window Size | Average Throughput | Retransmissions |
|---|---:|---:|---:|
| n7 → n11 | 4 KB | 722 Kbits/sec | 0 |
| n7 → n8 | 4 KB | 482 Kbits/sec | 0 |

### Interpretation

The longer path produced lower throughput because it introduced a higher Round Trip Time (RTT). TCP throughput is strongly affected by the relation:

```text
Throughput ≈ Window Size / RTT
```

With the same window size, a higher RTT means the sender waits longer for acknowledgements. Therefore, fewer useful bits are delivered per second.

This also demonstrates the effect of the **Bandwidth-Delay Product (BDP)**. If the TCP window is too small relative to the path delay, the sender cannot keep enough data in flight to fully utilize the network.

## 4. Capacity and Packet Loss Experiments

This part studied how TCP behaves when changing link capacity and packet loss rate. All tests used traffic from **n7 to n11** with a TCP window size of **4 KB**.

| Case | Capacity | Loss | Average Throughput | Retransmissions |
|---|---:|---|---:|---:|
| a | 10 Mbps | 0% | 723 Kbits/sec | 0 |
| b | 3 Mbps | 0% | 719 Kbits/sec | 0 |
| c | 10 Mbps | 5% | 271 Kbits/sec | 85 |
| d | 100 Mbps | 10% | 102 Kbits/sec | 57 |
| e | 10 Mbps | 1% on data path | 677 Kbits/sec | 10 |
| f | 10 Mbps | 1% on ACK path | 623 Kbits/sec | 21 |

### Main Findings

- A lower-capacity but loss-free link can perform better than a higher-capacity lossy link.
- Case **b** achieved high throughput despite having only **3 Mbps** capacity because there was no packet loss.
- Case **c** performed worse than case **b** because the 5% loss caused retransmissions and TCP congestion-control penalties.
- Case **d** performed the worst even though it had **100 Mbps** capacity because 10% packet loss severely reduced TCP performance.
- Loss on the ACK path caused more unnecessary retransmissions than loss on the data path.

### Conclusion

TCP performance depends not only on raw capacity but also on reliability, RTT, packet loss, and retransmission behavior. A clean and stable link can outperform a faster but lossy link.

## 5. OSPF Cost Change and Path Rerouting

In this experiment, a long `iperf3` session was started between **n7** and **n11**. Then `traceroute` was used to observe the active OSPF path before and after modifying interface costs.

### Default Path

Before any cost change, the path from **n7 to n11** was:

```text
n7 → 10.0.12.1 → 10.0.4.1 → 10.0.13.20 (n11)
```

The measured RTT was approximately:

```text
31.986 ms
```

### Cost Change on n5

The OSPF cost on `n5 eth1` was changed from **10** to **40**.

Example commands:

```bash
vtysh
show ip ospf interface eth1
configure terminal
interface eth1
ip ospf cost 40
exit
```

After changing the cost, OSPF recalculated the shortest path using Dijkstra's algorithm. The traffic was rerouted through an alternative path, and RTT increased to approximately:

```text
92 ms
```

### Interpretation

OSPF does not always choose the path with the highest raw bandwidth. It chooses the path with the lowest total OSPF cost. When the cost of an interface increases, OSPF recalculates the shortest path and updates the forwarding behavior dynamically.

---

## 6. Two Simultaneous TCP Connections

After restoring the OSPF cost on **n5** back to 10, two simultaneous TCP connections were started:

```text
n7 → n11
n11 → n7
```

Both connections achieved approximately:

```text
722 Kbits/sec
```

with zero retransmissions.

Then, the OSPF cost on `n4 eth1` was increased to **40**. The path changed, and RTT increased to about **92 ms**, but the throughput remained stable because the alternative path still had enough capacity.

### Interpretation

This confirms that OSPF can reroute traffic automatically without interrupting active TCP sessions, as long as the alternative route has sufficient capacity and stability.

## 7. OSPF Database and Link State Updates

The final experiment focused on OSPF control-plane behavior.

Wireshark was opened on **n2**, and the following filter was applied:

```text
ospf and not ospf.msg.hello
```

Initially, no non-Hello OSPF packets appeared, which confirmed that the network was stable and only periodic Hello packets were being exchanged.

### OSPF State Before Changes

The following commands were used on **n2**:

```bash
show ip ospf database
show ip ospf route
```

The routing table showed all reachable subnets through different next hops, including routes that passed through **n4**.

### OSPF Cost Change on n4

After changing the OSPF cost on `n4 eth1` to **20**, Wireshark captured multiple **LS Update (LSU)** packets.

This showed that OSPF flooded Link State Advertisements across the area so that all routers could update their Link State Databases and recompute routes.

### Bringing Down n4 Interfaces

After bringing down n4 interfaces, OSPF generated another wave of LS Update packets and LS Acknowledge packets. The larger LSU packets indicated that the topology-change information included multiple affected links.

After convergence, routes that previously used **n4** were updated to use alternative paths, and the OSPF database no longer contained active LSAs advertised by n4.

### Interpretation

This demonstrates OSPF's main strength: fast and automatic convergence after topology changes. OSPF uses LSAs, LSU flooding, acknowledgements, and SPF recalculation to maintain a consistent view of the network across all routers.

# Key Concepts Demonstrated

## TCP Window Size

The TCP window controls how much data can be in flight before waiting for acknowledgements. A very small window underutilizes the network, while an overly large window may overflow buffers and cause packet loss.

## Throughput

Throughput is the actual successful delivery rate of data. It may be much lower than the link capacity due to RTT, packet loss, retransmissions, congestion control, or small TCP window size.

## Retransmissions

Retransmissions occur when TCP believes packets or ACKs were lost. High retransmission counts usually indicate congestion, packet loss, or unstable network conditions.

## AIMD

AIMD stands for **Additive Increase, Multiplicative Decrease**. TCP increases its sending rate gradually but reduces it sharply when packet loss is detected.

## RTT and BDP

Round Trip Time affects TCP throughput because the sender often waits for acknowledgements. The Bandwidth-Delay Product helps determine how much data should be in flight to fully utilize the path.

## OSPF

OSPF is a link-state routing protocol. It uses interface costs and Dijkstra's algorithm to compute the shortest path. When link costs or topology change, OSPF floods LSAs and recomputes routes automatically.

# Final Conclusion

This project shows how TCP and OSPF behave in practical network scenarios. The experiments proved that TCP throughput is affected by window size, RTT, packet loss, retransmissions, and congestion control. They also showed that increasing capacity alone does not guarantee better performance if packet loss exists.

On the routing side, the project demonstrated how OSPF dynamically recalculates paths when interface costs or topology states change. Wireshark captures confirmed the role of LS Update and LS Acknowledge packets in propagating topology changes across the network.

Overall, the project connects theoretical networking topics such as TCP congestion control, throughput, RTT, BDP, packet loss, and OSPF link-state routing with real measurements from a virtual network lab.
